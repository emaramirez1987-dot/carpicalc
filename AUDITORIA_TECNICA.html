FASE 1 Árbol completo del proyecto
¿Qué es CarpiCalc? Una app web para carpinteros. Permite crear un catálogo de muebles con sus medidas y piezas, armar presupuestos, ver todo en 3D, calcular cuánto material se necesita y gestionar trabajos como un tablero Kanban. Funciona 100% en el navegador. Los datos se guardan en la nube (Supabase).

FASE 2 Arquitectura real implementada
Imaginá la app como una fábrica con departamentos: el Director (App.js) tiene todos los datos, el Almacén (storage.js) guarda y lee de la nube, los Calculistas (services/) hacen cálculos sin efectos secundarios, y los Trabajadores (components/) muestran la pantalla.

FASE 3 Mapa de componentes — los 15 archivos más importantes

1. App.js — El cerebro  CRITICIDAD MÁXIMA · 848 líneas

Componente raíz. Contiene autenticación, todo el estado de dominio, la cola de guardado serializada (withSave), el layout raíz con 10 pestañas, y carga todas las pestañas con React.lazy() excepto Presupuesto (siempre visible). Usa el patrón display:none para preservar el estado de pestañas visitadas sin desmontar el componente.

4. storage.js — Guardián de datos  CRITICIDAD ALTA · 364 líneas

Única capa que habla con Supabase y localStorage. Estrategia: delete-all + insert-all por workspace. Simple y confiable para un usuario. Las funciones más importantes: cargarDatos(), guardarModulos(), guardarPresupuestos(), cargarBorradorModulo().

5. state/NavContext.jsx — El navegador  CRITICIDAD ALTA · 120 líneas

Solo maneja navegación. Usa useReducer con 12 acciones semánticas. Patrón importante: las acciones son "puentes" — un componente pone datos, otro los consume y los borra (DEEPLINK_CONSUMIDO). Así no quedan estados colgados.

6. state/PresupuestoContext.jsx — El editor activo  CRITICIDAD ALTA · 43 líneas

Define solo el contexto. El estado real vive en AppInterna dentro de App.js. Expone: items, dimOverride, composicionOverride, inlineModulos, adicionales, costosDirectos, presupuestoActivoId.

7. visor3d/engine/buildPiezas3D.js — Motor 3D  CRITICIDAD ALTA

Función pura: módulo + costos + parámetros → lista de piezas con geometría 3D. No importa React ni Three.js. Proceso: expande con generarPiezas() → resuelve contexto → determina orientación por campo o por nombre → calcula d1/d2 → evalúa posFormulas → devuelve array de geometrías.

8. catalogo/index.jsx — El catálogo  CRITICIDAD ALTA · 1263 líneas

Exporta 3 componentes: CatalogoModulos (lista con búsqueda, import/export JSON, abre FormModulo), PanelSelectorModulos (selector compacto para el presupuesto), EditorVistaSVG (editor de composición visual del frente del módulo).

9. catalogo/FormModulo.jsx — Formulario de módulo  CRITICIDAD ALTA

El formulario más complejo. Tabs: Datos (nombre, dims, material) · Variables (VarsExplorer con carpetas) · Piezas (lista + FormPieza) · Herrajes · Avanzado (EditorParametrico) · Sub-componentes (EditorComponenteHijo). Preview SVG en tiempo real + barra de costo total.

10. presupuesto/index.jsx — Editor de presupuesto  CRITICIDAD ALTA

Gestiona todo el flujo: agregar/quitar módulos, editar dimensiones (AcordeonEdicionItem), ver SVG, guardar/cargar (GestorPresupuestos), exportar PDF. Indicador de autosave en tiempo real. Consume PresupuestoContext.

11. presupuesto/ConfiguradorParametrico.jsx — Controles paramétricos  CRITICIDAD MEDIA-ALTA

Un control por parámetro del módulo: number/integer → stepper +/- con debounce 150ms · boolean → toggle · choice → select · formula → solo lectura. Muestra warnings de constraints violadas en rojo.

12. presupuesto/GestorPresupuestos.jsx — Lista de presupuestos  CRITICIDAD MEDIA

Panel colapsable con todos los presupuestos. Calcula en tiempo real si alguno está desactualizado (compara total guardado vs. total calculado con precios actuales). Permite buscar, cambiar estado, cargar en editor, eliminar.

13. services/optimizerService.js — Optimizador de corte  CRITICIDAD MEDIA · 224 líneas

Algoritmo Guillotine 2D bin-packing mejorado. Ordena por área DESC, multi-placa open list, best-fit space, split BSSF. Calcula sobrantes reutilizables (≥300mm) y métricas: rendimiento %, desperdicio %.

14. catalogo/VarsExplorer.jsx — Explorador de variables  CRITICIDAD MEDIA

Muestra todas las variables disponibles en scopes jerárquicos: dims base → parámetros → vars derivadas de piezas → vars custom → vars de subcomponentes. Sistema de carpetas para organizar variables custom (metadatos separados, no afectan las fórmulas).

15. services/presupuestoService.js — Mutaciones de presupuesto  CRITICIDAD MEDIA · 156 líneas

Funciones puras: crearPresupuesto() (IDs con crypto.randomUUID()), eliminarPresupuesto() (limpia TEMP_ huérfanos), migrarDimOverridePresupuestos() (migra formato legacy), migrarTempEnPresupuestos() (TEMP_ → permanente).

FASE 4 Flujos reales del sistema

Flujo 1: Editar un módulo en el catálogo

1. Usuario clic "✎ Editar" en tarjeta del catálogo
   └─ CatalogoModulos → setModo({ tipo:"editar", codigo, modulo })

2. FormModulo se monta con los datos del módulo existente
   └─ Inicializa: datos, piezas, variables, herrajes, subComponentes, parametros

3. Usuario cambia dimensiones / agrega piezas
   └─ Estado local de FormModulo se actualiza
   └─ Preview SVG se re-renderiza en tiempo real (generarVistaSVG)
   └─ Costo total se recalcula (calcularModulo)

4. Usuario clic "💾 Guardar"
   └─ CatalogoModulos.guardar(codigo, datos)
   └─ setModulos({ ...modulos, [codigo]: datosNuevos })
   └─ onSave(todos) → App.js → withSave → guardarModulos(supabase)

5. Supabase recibe los datos nuevos (delete-all + insert-all)

Flujo 2: Armar un presupuesto

1. Usuario abre pestaña Presupuesto
   └─ Consume PresupuestoContext (items, dimOverride, etc.)

2. Busca un módulo en PanelSelectorModulos
   └─ Clic → agrega { id: uuid, codigo, cantidad: 1 } a items[]

3. Para cada ítem se muestra:
   └─ Preview SVG + costo calculado + botón "✎ editar"

4. Usuario clic "✎" en el ítem
   └─ AcordeonEdicionItem se expande
   └─ Sliders A/P/H + selector de material
   └─ Si el módulo tiene parámetros → ConfiguradorParametrico

5. Usuario clic "💾 Guardar presupuesto"
   └─ presupuestoService.crearPresupuesto() → ID UUID
   └─ App.js → withSave → guardarPresupuestos(supabase)

Flujo 3: Sistema paramétrico end-to-end

DISEÑO (catálogo):
  FormModulo → EditorParametrico
  └─ Carpintero define: cajones (integer, min 1, max 8)
  └─ Pieza "frente" con repeat: { var:"i", from:1, to:"cajones" }

USO (presupuesto):
  AcordeonEdicionItem → ConfiguradorParametrico
  └─ NumberStepper para "cajones" → usuario cambia a 5

CÁLCULO:
  calcularModulo(modulo, costos, { cajones: 5 })
  └─ resolverContextoModulo → modVars incluye cajones=5
  └─ generarPiezas → repeat expande frente × 5 piezas
  └─ Costo recalculado con 5 piezas

RENDER 3D:
  buildPiezas3D(modulo, costos, { cajones: 5 })
  └─ 5 piezas de frente posicionadas con "i" en posFormulas

Flujo 4: Edición Nivel 3 (editar módulo desde el presupuesto)

1. "🔧 Editar en catálogo" en un ítem del presupuesto
   └─ crearTempDesdeModulo() → copia con código TEMP_1234567890
   └─ dispatch(INICIAR_EDICION_NIVEL3)

2. NavContext → catalogoDeepLink = tempCod
   └─ Vista cambia a "catalogo"
   └─ CatalogoModulos abre FormModulo con el TEMP_

3. Usuario edita → cambios solo afectan la copia temporal

4a. Guardar "solo para este presupuesto":
    └─ TEMP_ queda como módulo temporal vinculado al presupuesto

4b. Guardar "también en catálogo":
    └─ Nuevo ID permanente: MC${Date.now().slice(-6)}
    └─ TEMP_ eliminado + referencias actualizadas en items[]

5. dispatch(VOLVER_A_PRESUPUESTO) → vuelve al presupuesto

Flujo 5: Cola serializada de guardado (withSave)

// El problema: cambios rápidos → múltiples saves paralelos
// → race conditions → datos corruptos.

// La solución:
withSave(async () => await guardarModulos(costos))

// Internamente:
// 1. Agrega la función a saveQueue[]
// 2. Si no hay save corriendo → llama a drain()
// 3. drain() ejecuta uno por uno en orden
// → Nunca dos saves corren al mismo tiempo

Flujo 6: Detección de precios desactualizados

Carpintero sube el precio del material
→ guardarCostos() → bumps costos_version en localStorage

GestorPresupuestos al renderizar:
→ recalcularTotalPresupuesto(p, modulos, costos) por cada presupuesto
→ |calculado - guardado| > $1 → "⚠ Desactualizado"

Usuario clic "Actualizar":
→ Se guarda el nuevo total en Supabase

No existen dependencias circulares. El grafo de importaciones es un árbol dirigido sin ciclos (DAG). Fue un diseño intencional. utils.js no importa nada del proyecto. moduloService.js solo importa utils.js y constants.js.

Puntos de acoplamiento fuerte

  1. resolverContextoModulo — llamada desde 4 lugares (buildPiezas3D, calcularModulo, _contextoParametrico, VarsExplorer). Cambiar su firma impacta todos.
  2. El shape del objeto modulo — todos los componentes esperan el mismo contrato. Agregar un campo requiere actualizar MODULO_VACIO y parsearModulo.
  3. dimOverride key format — la clave es item.id || item.codigo. Cambiarlo rompe overrides existentes (hay migración implementada).
  4. withSave en App.js — todos los saves pasan por esta cola. Si falla, ningún save funciona.
FASE 7 Puntos críticos actuales

🔴 Riesgos altos

1. modulo.variables shape inconsistente

parsearModulo valida variables como Array. Pero resolverVariables en utils.js lo trata como objeto { nombre: formula }. Y el formulario también lo trata como objeto. En la práctica funciona porque la conversión ocurre implícitamente, pero es una bomba de tiempo si se cambia la serialización. Documentado en ARCHITECTURE.md como deuda técnica pendiente.

2. App.js con 848 líneas y creciendo

Es el componente más grande. Está bien organizado pero cualquier funcionalidad nueva tiene la tentación de ir ahí. Si crece más se vuelve difícil de mantener.

🟡 Riesgos medios

3. Módulos TEMP_ conviven con permanentes en el mismo objeto

Están filtrados en PanelSelectorModulos y CatalogoModulos, pero puede filtrarse si se agrega un nuevo punto de acceso al catálogo.

4. Borrador de presupuesto puede quedar desincronizado

Si el usuario abre la app en dos pestañas del navegador al mismo tiempo, el borrador en localStorage se pisa. Limitación documentada de app single-user.

🟢 Riesgos bajos (ya manejados)

5. Fórmulas con new Function() — seguridad

Tiene whitelist de palabras permitidas. Solo el carpintero autenticado puede definir fórmulas para su propio uso. Riesgo bajo en este contexto.

6. Delete-all + insert-all

Window de milisegundos entre delete e insert. Supabase RLS garantiza que nadie más puede escribir en ese workspace. Riesgo muy bajo pero documentado.

7. Imágenes como base64

Las imágenes del catálogo se guardan como base64 en el objeto módulo. Hay compresión implementada (comprimirImagen) pero no hay límite de tamaño forzado.

FASE 9 Resumen ejecutivo
El dato más importante de toda la arquitectura: hay una única función que todo el sistema usa para calcular fórmulas: resolverContextoModulo() en services/moduloService.js. Fue creada para reemplazar tres implementaciones paralelas con bugs distintos. Hoy, si una fórmula funciona bien, funciona igual en el presupuesto, el 3D, el optimizador de corte y el PDF.

Fortalezas del proyecto

  1. Arquitectura en capas clara y respetada — cada archivo tiene una sola responsabilidad bien definida.
  2. Función de resolución centralizada — resolverContextoModulo evita la duplicación del motor de fórmulas.
  3. Parser que nunca rompe — parsearModulo garantiza que datos viejos o incompletos nunca crashean la app.
  4. Cola de guardado serializada — withSave previene race conditions sin bibliotecas externas.
  5. Sistema paramétrico completo — repeat, condition, constraints, zonas, subComponentes.
  6. Tests automáticos — moduloService.test.js y buildPiezas3D.test.js cubren las funciones más críticas.
  7. Zero dependencias circulares — el grafo de importaciones es un árbol limpio.

Para hacer cambios grandes con seguridad:
1. Correr los tests antes y después de cambiar utils.js o moduloService.js
2. Usar resolverContextoModulo para cualquier código nuevo que evalúe fórmulas
